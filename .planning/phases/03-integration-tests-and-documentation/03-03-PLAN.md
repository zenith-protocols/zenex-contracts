Create test-suites/tests/test_fee_accrual.rs (TEST-04).

**Funding conservation test design (addresses review concern about conflating fee channels):**
The key concern was that funding "zero-sum" may not hold exactly if borrowing, protocol fees, etc. are also active. Solution: design the funding test to ISOLATE funding from other fee channels.

Required tests:

Funding conservation (zero-sum per D-10):
1. `test_funding_is_zero_sum` -- ISOLATED setup:
   - Open equal-sized long and short positions (same notional)
   - This means neither side is "dominant" for borrowing, AND funding rate is 0 with balanced sides
   - To actually test funding flow: open unequal sides (e.g., 2x long vs 1x short), so funding flows from long to short
   - Use apply_funding, jump 1 hour, close both
   - Key assertion: `user_long_delta + user_short_delta + vault_delta + treasury_delta == 0` within rounding tolerance of 2 units
   - Total = sum of: each user token balance, vault total_assets, treasury token balance
   - NOTE: borrowing is also active for the dominant side. So the conservation is total system tokens, not just funding channel. This is correct: the test proves the SYSTEM is zero-sum (no tokens created/destroyed), which subsumes funding conservation.

2. `test_funding_dominant_side_pays` -- open $10k long and $5k short (long is dominant), jump 1 hour, call apply_funding, then close both. Verify: long user's payout is reduced by funding, short user's payout is increased. Assert: `long_funding_paid > 0` and `short_funding_received > 0` (direction check).

3. `test_funding_rate_zero_with_balanced_sides` -- open equal long and short notionals, call apply_funding, verify `get_market_data().fund_rate == 0`

Borrowing curve accuracy (TEST-04):
4. `test_borrowing_curve_at_utilization_points` -- For utilization levels (25%, 50%, 75%, 90%):
   - Deposit controlled vault balance (e.g., 100_000_0000000)
   - Open positions to reach target utilization (total_notional / vault_balance)
   - Jump 1 hour to trigger accrual
   - Read `get_market_data()` to get `borr_rate`
   - Compute expected rate OFF-CHAIN: `r_base * (1 + r_var * util^5)` using values from `default_config()` in trading/src/testutils.rs
   - Assert: `observed_rate approx_eq expected_rate` with 1% relative tolerance
   - This tests the FORMULA, not just that some rate exists
   NOTE: Read `default_config()` first to get the actual r_base and r_var values. The off-chain computation must use the same values.

5. `test_borrowing_dominant_side_only` -- open positions where long notional > short notional (long is dominant), jump 1 hour. Read market_data: assert `l_borr_idx > initial_l_borr_idx` (long borrowing advanced) and `s_borr_idx == initial_s_borr_idx` (short borrowing unchanged). This directly proves dominant-side-only borrowing.

Fee accrual over time:
6. `test_fee_accrual_increases_with_time` -- open position, jump 1 hour, get_position to check snapshot, jump another hour, close position, verify total fees are larger than after 1 hour alone. (Proves accrual is time-proportional.)

Token conservation invariant:
7. `test_token_conservation_after_open_close` -- record total tokens across all addresses (users, vault, treasury, trading contract) before any trading. Open several positions, close them all. Record total tokens after. Assert: `total_before == total_after` within rounding tolerance (2 units per position). This is the deterministic version of the existing proptest invariant.
