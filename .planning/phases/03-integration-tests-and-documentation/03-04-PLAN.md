**Auth harness design (addresses review concern about strategy):**

Two deployment approaches based on contract type:

A) **Fixture-based** (for trading + vault that need full stack): Use TestFixture with `mock_all_auths_allowing_non_root_auth()`. This means:
   - Root-level auths (the direct caller's auth) are NOT mocked
   - Sub-contract auths (cross-contract calls within a transaction) ARE mocked
   - So admin functions (`#[only_owner]`) and user functions (`user.require_auth()`) naturally enforce real auth at the root level

B) **Direct deployment** (for price-verifier, treasury, timelock, factory): Deploy via `e.register()` WITHOUT `mock_all_auths()`. These are simpler contracts that don't need the full stack. Use `e.mock_auths(&[MockAuth { address: &non_owner, ... }])` to provide auth for the wrong caller.

**For each test, the pattern is:**
1. Deploy the contract (either via fixture or directly)
2. Create `non_owner = Address::generate(&e)`
3. Mock auth for `non_owner` (not the real owner)
4. Call `client.try_function_name(...)` with the non_owner's auth
5. Assert `result.is_err()` -- the call failed because auth check rejected the non_owner

**Required tests (20 auth call sites):**

**Two error types to distinguish (addresses review concern):**
- "unauthorized caller" = auth check fails (wrong address provided auth)
- "wrong stored owner/strategy" = caller is authed but is not the expected owner/strategy in storage

For #[only_owner] functions, both checks are folded into one: the macro checks caller == stored owner. If a non-owner provides auth, the `require_auth()` check on the stored owner address fails (because non_owner != stored owner).

For strategy_withdraw, there's an additional check: strategy == stored_strategy. So even a fully-authed random address fails because it's not the strategy.

**Important implementation detail:** For fixture-based auth tests (trading/vault), the TestFixture::create() already calls `mock_all_auths_allowing_non_root_auth()`. This is correct for auth testing because:
- Admin functions: the #[only_owner] macro calls `owner.require_auth()` at root level, which is NOT mocked. So calling set_config when you're not the owner will fail because the env has no root-level auth for the owner address -- only the non_owner's auth was mocked.
- User functions: similarly, `user.require_auth()` is at root level.

If the above doesn't work as expected (i.e., mock_all_auths_allowing_non_root_auth still allows the calls), fall back to creating an `Env::default()` WITHOUT any auth mocking, and use `e.mock_auths()` precisely for each test.

Create test-suites/tests/test_timelock.rs. These tests exercise the generic timelock with real target contracts (per D-02).

**Timelock setup pattern (addresses review concern about underspecified ownership):**

Each test must explicitly:
1. Deploy the full stack via TestFixture (which creates trading, vault, timelock)
2. Transfer trading contract ownership to the timelock: call `trading.transfer_ownership(timelock_address)` (or the equivalent -- check trading/src/contract.rs for the ownership transfer mechanism; if it uses stellar-access, it may be `set_owner` or similar)
3. Now the timelock is the owner of the trading contract
4. Queue calls on the timelock with `target = trading_address`
5. Jump past delay
6. Execute

NOTE: If trading does NOT have a transfer_ownership function, then the TestFixture must be modified to deploy trading with the timelock as the initial owner. Check the stellar-access crate's Ownable trait for the available methods. In that case, the factory deployment must set `admin = timelock_address`.

**Required tests:**

Queue/Execute flow:
1. `test_timelock_queue_and_execute_set_config` -- set timelock as trading owner, queue a set_config call with modified TradingConfig (e.g., change r_base), jump past delay (e.g., fixture uses delay=86400, so jump 86401), execute, verify `trading.get_config()` returns the new config with the changed r_base value
2. `test_timelock_queue_and_execute_set_market` -- similar for set_market with a MarketConfig change (e.g., change max_util)
3. `test_timelock_execute_before_delay_reverts` -- queue a call, try execute BEFORE delay passes (jump only half), expect TimelockError::NotUnlocked error
4. `test_timelock_cancel_prevents_execution` -- queue a call, cancel it, try execute after delay, expect TimelockError::NotQueued error
5. `test_timelock_set_status_immediate` -- set_status bypasses delay (emergency halt). Call timelock.set_status(trading_address, 1) (OnIce), verify `trading.get_status() == 1` immediately without waiting for delay.
6. `test_timelock_set_delay_through_timelock` -- set_delay queues the delay change with the CURRENT delay. Then call apply_delay after the current delay passes. Verify the new delay is now active.
7. `test_timelock_execute_after_cancel_reverts` -- per T-ELEV-12 governance front-run concern: queue, cancel, jump past delay, try execute, verify failure. (This may overlap with test 4, but specifically addresses the threat model scenario.)
